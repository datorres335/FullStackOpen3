npm install graphql-ws ws @graphql-tools/schema
	// these three libraries usually show up when you're implementing GraphQL subscriptions (real-time features) manually.
	// you’re preparing your server to support:
		WebSockets
		Real-time GraphQL subscriptions
		Custom schema creation
	// graphql-ws: Implements the GraphQL over WebSocket protocol. It allows clients to:
		Open a persistent WebSocket connection
		Subscribe to real-time GraphQL events
		Receive pushed updates from the server
	// ws: WebSocket server implementation for Node.js. ws provides the actual:
		WebSocket server
		TCP handling
		Low-level socket management
	// @graphql-tools/schema: Creates an executable GraphQL schema from:
		Type definitions (SDL)
		Resolver functions
		It’s used when you aren’t relying on a framework (like NestJS) to generate the schema for you.
	// When you build a custom GraphQL server (especially with Apollo Server v4):
	You need:
		@graphql-tools/schema → create the schema
		ws → create WebSocket server
		graphql-ws → run subscriptions over that WebSocket
	Together they enable:
		GraphQL Queries (HTTP)
	GraphQL Mutations (HTTP)
	GraphQL Subscriptions (WebSocket)

npm install graphql-subscriptions
	// It provides a simple Publish/Subscribe (PubSub) system for GraphQL subscriptions.
	// It helps your server:
		Publish events when something happens (e.g., message created)
		Notify all subscribed clients
		Wire those events into GraphQL subscription resolvers
		- It does not create WebSockets.
		- It does not run the server.
		- It only manages event broadcasting.
	🎯 Important Limitation
	The default PubSub implementation is 'In-memory only'
	Meaning:
		Works only within one server instance
		Does NOT scale horizontally
		Not production-safe for distributed systems
	In real production systems, people replace the default PubSub with:
		Redis PubSub
		Kafka
		RabbitMQ
		Google PubSub
		AWS SNS/SQS
	Apollo even recommends Redis for scaling subscriptions, AKA:
		'graphql-redis-subscriptions'

Docker Command Hierarchy Tree:
docker
  ├── container
  │      ├── run        // create and start a new container from an image
  │      ├── ls         // list containers (running or all with -a)
  │      ├── rm         // remove one or more containers
  │      ├── inspect    // show detailed info about a container
  │      ├── start      // start an existing, stopped container
  │      ├── stop       // gracefully stop a running container
  │      ├── kill       // immediately stop a running container (SIGKILL)
  │      ├── logs       // view logs from a container
  │      ├── prune      // remove all stopped containers
  │      └── exec       // run a command inside a running container
  ├── image
  │      ├── build      // build an image from a Dockerfile
  │      ├── ls         // list images stored locally
  │      ├── rm         // remove one or more images
  │      ├── inspect    // show detailed info about an image
  │      ├── prune      // remove unused images
  │      ├── commit     // create a new image from a container's current state
  │      └── pull       // download an image from Docker Hub
  ├── volume
  │      ├── ls         // list volumes
  │      ├── rm         // remove a volume
  │      ├── inspect    // show detailed info about a volume
  │      ├── create     // create a new volume
  │      ├── prune      // remove unused volumes
  │      └── update     // modify volume configuration
  ├── network
  │      ├── ls         // list networks
  │      ├── rm         // remove a network
  │      ├── inspect    // show detailed info about a network
  │      ├── create     // create a new network
  │      ├── connect    // attach a container to a network
  │      ├── disconnect // detach a container from a network
  │      ├── prune      // remove unused networks
  │      └── update     // modify network configuration
  └── compose
         ├── up         // start all services defined in docker-compose.yml
         ├── down       // stop and remove all services, networks, volumes
         ├── build      // build images defined in the compose file
         ├── logs       // view logs from all services
         ├── ps         // list running services
         ├── exec       // run a command in a running service container
         ├── restart    // restart all or specific services
         └── stop       // stop services without removing them

- container: manages one running instance
- image: manages templates
- volume: manages persistent data
- network: manages connectivity
- compose: manages all of the above together as a single application; multi‑container orchestration (define and run whole app stacks)

QUESTION: 
	Are containers usually used for microservices architecture apps?
ANSWER:
	Containers are a natural fit for microservices, and most modern microservice architectures use them, but it’s not a strict requirement. 		The relationship is strong because containers solve several problems that microservices create.
	- Why containers and microservices pair so well:
	Microservices break an application into many small, independently deployable services. That creates challenges around consistency, 			isolation, deployment, scaling, and dependency management. Containers directly address those challenges.
	- When containers might not be necessary:
	A monolithic application or a small project with only one or two services may not need containers at all. Containers add operational complexity, 	so they’re most beneficial when scaling or isolating services matters.
